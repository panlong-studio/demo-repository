# 🚀 团队 Git 开发标准化工作流 (v1.0)

> **💡 核心理念：** 保护 `main` 分支，所有改动走 **PR (Pull Request)**。保持历史线性。

------

### 📂 1. 开启新任务 (每日开工必做)

这一仓库的要求中是同步长期的个人分支，即按A操作
之后的项目仓库中按功能建分支 不再长期维护个人分支，即按B操作

#### 路径 A：同步长期的个人分支

*如果你当前仍在使用旧的长期分支，先将其对齐主干：*

```
git checkout main             # 1. 切换到主分支
git pull origin main          # 2. 拉取远程最新进展
git checkout <你的长期分支名>    # 3. 回到自己的长期分支
git merge main                # 4. 将主分支新进展合入
```

#### 路径 B：按功能新建分支 (推荐模式)

*新项目或新功能建议直接采用此方式，开发完即删：*

```
git checkout main             # 1. 确保在主分支
git pull origin main          # 2. 获取主干最新代码
git checkout -b <姓名-功能>    # 3. 创建并切换新分支 (例: ywj-login-api)
```

------

### 💻 2. 开发与本地提交

在功能分支上进行小步快跑式的开发：

```
# ... 编写/修改代码 ...

git add .                     # 1. 将所有改动放入暂存区
git commit -m "feat: 描述信息" # 2. 提交到本地 (例: feat: 增加用户登录校验)
```

------

### 🔄 3. 保持分支同步 (防冲突关键)

在开发中或准备提交 PR 前，**必须**同步远程主干，确保你的代码基于最新版本：

```
git fetch origin              # 1. 抓取远程最新状态
git rebase origin/main        # 2. 以变基方式同步，保持历史线性整洁
```

------

### 📤 4. 推送并申请合并

```
git push origin <你的功能分支名> 
# 推送成功后，去 GitHub 网页端点击 "Compare & pull request" 发起合并申请
```

------

### 🛠️ 5. 冲突处理 (Rebase 报错时)

如果同步时出现 `CONFLICT` 字样，不要慌，按以下步骤处理：

1. **手动排查**：打开编辑器，搜索 `<<<<<<<`，删除符号并保留正确的代码代码。
2. **标记解决**：执行 `git add <冲突文件名>`。
3. **继续流程**：执行 `git rebase --continue`。
4. **撤销重来**：如果彻底搞乱了，想回到 Rebase 前，执行 `git rebase --abort`。

------

### 🚫 ⚠️ 团队禁忌事项 (Red Lines)

> [!CAUTION]
>
> 1. **严禁直推**：严禁在 `main` 分支直接执行 `git push`。
> 2. **严禁强推**：严禁在 `main` 分支执行 `git push --force` (会覆盖全组代码历史)。
> 3. **隐私安全**：敏感信息 (服务器密码、API Key) 严禁上传至仓库。
> 4. **无用文件**：编译产生的二进制文件 (`.exe`, `.obj`) 严禁提交。
